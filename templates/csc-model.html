CSC2 · model training pipeline
End-to-end documentation of how the CSC2 forecast-correction model is built — from raw EURO/GFS forecast cycles and NDBC/CDIP buoy observations through pairing, feature engineering, fitting, evaluation, and registry surfacing.

1 · Anatomy of the data

Forecast cycle shards

Every 00Z and 12Z run for both EURO and GFS is captured by csc2.logger (launchd com.colesurfs.csc2-log, 3 AM + 3 PM ET) and persisted as one parquet file per cycle, partitioned by model and buoy:

.csc2_data/forecasts/model={EURO,GFS}/buoy=<id>/year=Y/month=M/cycle=YYYYMMDDTHHZ.parquet

  • EURO rows come from waves_cmems.fetch_cmems_point (CMEMS ANFC) called by csc2.logger.
  • GFS rows come from waves.fetch_wave_forecast(model_key="GFS") against Open-Meteo (and from csc2.aws_gfs_backfill for the historical fill).
  • The on-disk schema is defined in csc2/schema.py as FORECAST_COLUMNS — raw partition fields (sw1_*, sw2_*) plus combined-sea fields (combined_*) so the dashboard's v1.7.1 GFS fallback can be reconstituted at read time.
  • Every cycle preserves the full +0…+240 h lead structure.

Buoy-observation shards

Persisted under .csc_data/observations/buoy=<id>/year=Y/, split by partition (matching the dashboard's spectral decomposition):

FilePartitionSourceNotes
stdmet*.parquet0 (combined sea)NDBC text archivesNot used as a training target — partition=0 doesn't match the dashboard quantity the model predicts.
spectral*.parquet1 / 2 (primary / secondary swell)buoy._spectral_componentsThe trainable obs targets.

Spectral data is sourced in fallback order:

  1. NDBC realtime spectral — most recent ~45 days (data_spec + swdir).
  2. NDBC historical swden + swdir yearly archives — only available for the subset of NDBC-archived stations.
  3. CDIP historic.nc + realtime.nc — fills in stations NDBC doesn't archive.

Unified obs schema: buoy_id, valid_utc, partition, hs_m, tp_s, dp_deg, source, ingest_utc.

Live observation logger

csc2.obs_logger (launchd com.colesurfs.csc2-obs, every 30 minutes) appends the freshest NDBC reading to .csc_data/live_log/observations/, deduped on (valid_utc, partition). This is what closes the loop on in-progress cycles whose forecasts are already on disk but whose verifying obs haven't arrived yet.

2 · Processing pathway

Pairing logic

Implemented in csc2/archive_status.py. For each (buoy, cycle), the cycle is considered paired only if ≥ 95 % of the shared EURO ∩ GFS forecast hours have a partition=1 OR partition=2 observation snapped to the hour. The 95 % threshold tolerates routine single-hour NDBC dropouts without letting cycles with material buoy outages slip into training.

Coverage figure
The coverage token baked into model names is computed as len(histograms.combined_east.paired_by_doy) / 365 — i.e. the count of unique calendar dates (uncollapsed across years) on which at least one east-pool buoy clears the 95 % bar, divided by 365 (always 365, never 366). Once we cross into year 2 this fraction can exceed 1.0.

Trainer load

csc2/train.py:build_paired_dataset performs an inner-join of EURO + GFS on the tuple (buoy_id, cycle_utc, valid_utc, lead_hours), then left-joins the obs partition=1 and partition=2 shards wide so each row carries observed sw1 and sw2 alongside the two model forecasts.

_apply_dashboard_fallback_gfs (at csc2/train.py:106) synthesizes GFS sw1 from combined_* fields when partitions are absent — byte-identical to the dashboard's v1.7.1 fallback in waves.py:_parse_response. Rows where both obs sw1 and sw2 are missing are dropped.

Time-aware holdout split

The last 15 % of unique cycles, sorted by cycle_utc, form the test set. Per-cycle (rather than per-row) splitting prevents lead-hour leakage where a same-cycle row could appear on both sides of the split.

Feature engineering

csc2/train.py:add_features derives:

  • sin/cos of EURO and GFS sw1 + sw2 directions (avoids the 0°/360° wraparound).
  • EURO − GFS deltas for sw1/sw2 height, period, and circular-direction difference.
  • sin/cos day-of-year — captures seasonal swell-regime structure.
  • Buoy one-hot (5 east-coast buoys) — lets the shared model learn per-buoy splits.
  • lead_hours as a first-class feature — every cycle preserves the full +0…+240 h lead window.

3 · Model architectures

CSC2+baseline

Per-(buoy × lead_hour × variable) additive bias on top of the EURO/GFS arithmetic mean. Deliberately simple — no learned interactions, no time-of-year, no EURO–GFS delta. This is the floor the ML model must beat. Fully customised per buoy.

For Hs / Tp: pred = mean(EURO, GFS) + bias[buoy, lead, var] For Dp: base = circular_mean(EURO, GFS) pred = base + bias_deg[buoy, lead] (mod 360)

Direction uses circular mean via sin/cos averaging plus a per-cell rotation bias.

CSC2+ML

LightGBM gradient-boosted trees, 8 boosters — one per scalar target. Buoy is supplied as a one-hot feature so the same shared model learns per-buoy splits. Direction is recombined via atan2(sin, cos) at inference time.

  1. sw1_height_ft
  2. sw1_period_s
  3. sw1_dp_sin
  4. sw1_dp_cos
  5. sw2_height_ft
  6. sw2_period_s
  7. sw2_dp_sin
  8. sw2_dp_cos

Hyperparameters

learning_rate = 0.05 num_leaves = 63 num_boost_round = 400 feature_fraction = 0.9 bagging_fraction = 0.9 bagging_freq = 5 seed = 42 deterministic = True

4 · Evaluation metrics

Traditional

Per-variable MAE / RMSE / bias across the six dashboard variables (sw1/sw2 × Hs/Tp/Dp), reported on each model's own time-aware holdout. Surfaced in Table 1 on /csc.

Surfer

Sensitivity / specificity / PPV / NPV per swell category — FUN, SOLID, FIRING, HECTIC, MONSTRO, plus the combined FUN-OR-BETTER positive class — using swell_rules.categorize applied to (predicted sw1 Hs, sw1 Tp). This is the same categoriser the dashboard uses for cell coloring, so the metric reflects the surfability call the user actually sees.

Composite skill (registry)

A single weighted-sum score per model, used to pick the TOP slot in the active-models table. Each per-variable MAE skill is 1 − model_MAE / euro_MAE; the surfer F1 skill is (model_F1 − euro_F1) / (1 − euro_F1) on FUN-OR-BETTER. Positive values beat raw EURO on average.

sw1 height0.25
sw1 period0.25
sw1 direction0.05

surfer F1 (FUN-OR-BETTER)0.30

Only models whose holdout has ≥ 1000 rows are eligible for the TOP slot — small-holdout skill numbers are too noisy to rank against year-scale evaluations.

5 · Naming convention

Every trained model is tagged:

CSC2+{baseline|ML}_{YYMMDD}_{coverage}_v{N}
  • YYMMDD — train date in UTC, sorts lexicographically (e.g. 260424).
  • coverage — east-pool fraction of 365 paired calendar dates, rounded to 0.01.
  • vN — same-day variant; bump for any structural change (different feature set, different LightGBM params, different baseline binning).

Weights land in .csc2_models/east/<full-name>/. The west track uses the identical convention under .csc2_models/west/<full-name>/ and never surfaces on the dashboard until explicitly promoted.

Examples: CSC2+baseline_260424_0.77_v2, CSC2+ML_260424_0.77_v2.

Dashboard / CSC2 identity invariant

Every CSC2 forecast row must match the dashboard byte-for-byte for the same (buoy, valid_utc, model) tuple. Anything that feeds training passes through waves_cmems / waves exactly the way the live dashboard does — no shortcut pulls of raw CMEMS or raw GRIB values. If CSC2 output disagrees with a dashboard cell for the same hour/buoy, something is wrong — not a model difference.
